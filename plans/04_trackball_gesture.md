# 04. トラックボール方向ジェスチャー（キー＋方向→アクション）

特定のキーを押しながらトラックボールを上下左右に振ると、その方向に応じたアクション（任意のキー／ビヘイビア）を発火する機能を実装しました。アクションの割り当てはすべて `config/keymap.keymap` 側で行えるため、C のコードを再コンパイルせずに変更できます。

---

## 課題

ZMK の入力プロセッサ（`zip_*`）や公式の Behaviors Input Processor は、トラックボールの **移動方向（上下左右）を判定して任意のアクションへ変換する** 機能を持ちません（Behaviors Input Processor はバイナリの `INPUT_EV_KEY` 向けで、相対移動ベクトルには非対応）。方向→アクションを行う既製のコミュニティモジュールも存在しませんでした。

そのため、以下を満たす独自の仕組みが必要でした。

* 「あるキーを押しながら」という条件付きで発火する
* 押すキーの種類ごとに別のアクションを割り当てられる
* アクションの割り当てはキーマップ側で柔軟に変更できる

---

## 解決策（仮想キーポジション方式）

### 1. ジェスチャー検出 C コード `src/trackball_gesture.c`

`src/board.c` と同じく `INPUT_CALLBACK_DEFINE(NULL, ...)`（central 限定）でトラックボールの **生の X/Y 変位** を購読します。入力コールバックはリスナーの入力プロセッサより前に発火するため、カーソル用の変換（反転・スケール・スクロール化）に影響されず常に生値を取得できます。

ジェスチャー有効レイヤーがアクティブな間だけ X/Y を蓄積し、優勢軸が閾値を超えたら方向を確定して、4 つの **仮想キーポジション** を `raise_zmk_position_state_changed()` で press → （遅延後）release します。

主な調整パラメータ（ファイル冒頭の `#define`）:

* `GESTURE_THRESHOLD`（既定 80）: 発火に必要な蓄積変位量
* `GESTURE_IDLE_REARM_MS`（既定 300）: ボールがこの ms 間アイドルになると再武装する。連続した1モーションにつき1回だけ発火させ、連射を防ぐための静止時間（旧 `GESTURE_COOLDOWN_MS` を置換）
* `GESTURE_RELEASE_DELAY_MS`（既定 15）: press から release までの間隔
* `gesture_layers[]`（既定 `{9}`）: ジェスチャーを有効にするレイヤー番号の配列
* `GESTURE_POS_UP/DOWN/LEFT/RIGHT`: 仮想ポジション番号（dtsi の追加順と一致させる）

### 2. 仮想ポジションの確保（`torabo_tsuki_lp.dtsi`）

`size_l_transform` の `map` 末尾に、物理的に配線されていない `RC()` を 4 つ追加しました。マトリクススキャンからは決して発火せず、ジェスチャーハンドラからのイベントだけがこの位置を駆動します。

```dts
// 66=UP, 67=DOWN, 68=LEFT, 69=RIGHT
RC(0,6) RC(1,6) RC(0,13) RC(1,13)
```

ZMK は「最大レイアウト（ここでは L）にキーマップ長を合わせる」仕様のため、これに合わせて **全レイヤーのバインド数を 66 → 70** に拡張しました（各レイヤー末尾に 4 セル追加）。

さらに **`physical_layout_l` の `keys`（物理キー定義）にも `key_physical_attrs` を 4 つ追加**して 70 に揃えます。理由: ZMK はジェスチャーハンドラが raise したポジションを **物理レイアウトの position map 経由で変換**しており、`位置 >= マップ長`（マップ長 = 物理レイアウトのキー数）のイベントは破棄されるためです。

> **【重要・実装の最終形】** 上記の「仮想ポジション 66–69」方式は、トランスフォーム・物理レイアウト・全レイヤーを 70 に揃えても**実機で発火しなかった**（複数物理レイアウト構成の position map 解決を通らずイベントが破棄された）。そこで最終的に、**発火先を実在する未使用ポジション 0–3**（トップ行＝全レイヤー `&none`／物理未実装）に変更した。これは通常キーと同じ経路で確実にルーティングされる。`Trackball_Gesture`（Layer 9）の**先頭4バインド**を UP/DOWN/LEFT/RIGHT に割り当て、`src/trackball_gesture.c` の `GESTURE_POS_*` は `0,1,2,3` を指す。上記の 70 拡張（トランスフォーム／物理レイアウト／各レイヤー末尾4セル）は名残として残っているが未使用で、整理する場合は 66 に戻してよい。

### 3. 「キーを押しながら」＝モーメンタリレイヤー

既存の `&mo`（モーメンタリレイヤー）パターンをそのまま流用します。任意のキーに `&mo 9` を割り当てると、押している間だけ `Trackball_Gesture`（Layer 9）が有効になり、その間のジェスチャーがこのレイヤーの仮想ポジションのバインドにルーティングされます。押すキー（＝遷移先レイヤー）を分ければ、キーごとに別アクションを割り当てられます。

### 4. ジェスチャー中のカーソル抑制（不採用・要再設計）

当初は Layer 9 で `zip_xy_scaler 0` を適用してカーソルを止める `gesture_suppress` を入れていたが、**これを入れると検出が一切発火しなくなる**問題が実機で判明し、削除した。

理由: Zephyr の入力コールバックは**同一の `struct input_event` 実体を登録順に各コールバックへ渡す**。ZMK の input-listener（カーソル処理）が検出コールバックより**先**に走り、`zip_xy_scaler 0` が `evt->value` を 0 に上書きするため、後段の検出コールバックは常に 0 を読み、蓄積が閾値に届かず発火しない（「カーソルは止まる＝scaler 0 は効いているのにキーが出ない」という症状になる）。

したがってカーソル抑制を入れ直すには、検出を**リスナーより前**で行う必要がある（例: 検出自体をカスタム input-processor として `&pointing_listener` の先頭に挿す等）。現状はカーソルが動いたまま。

---

## 設定方法

1. トリガーキーを割り当てる。本構成では左親指の `&lt 9 BACKSPACE`（タップ=BACKSPACE／ホールド=ジェスチャー有効）を採用。トラックボールを右親指で操作するため、反対の左親指をトリガーにしている。元の `&lt 4 BACKSPACE` のホールド先 Layer 4 は全 `&trans` で未使用だったため振り替えても機能損失はない。別キーにする場合は `&mo 9` か `&lt 9 <tap>` を使う。
2. `Trackball_Gesture` レイヤー末尾の 4 バインド（UP / DOWN / LEFT / RIGHT）を好みのアクションへ書き換える（既定は矢印キー）。
3. 押すキーごとに別アクションにしたい場合は、レイヤーを追加し、`src/trackball_gesture.c` の `gesture_layers[]` にそのレイヤー番号を足して再ビルドする。
4. 方向が逆に感じる場合は `GESTURE_POS_*` の対応、または `zip_xy_transform` の反転設定に合わせて入れ替える。

---

## 効果と検証

* キーを押しながらトラックボールを振るだけで方向別アクションが発火し、押すキーを変えればアクションも変わる、という直感的な操作を実現。
* アクションの割り当ては通常のキーマップ編集（keymap-editor / zmk-studio でも可）で完結し、C の再コンパイルは不要。
* 閾値・クールダウンで誤爆と過剰なリピートを抑制。ボールを振り続ければ一定間隔で自動リピートも可能。

> 注: ビルドは GitHub Actions（CI）でのみ行われるため、実機での発火閾値・方向は書き込み後に微調整してください。
