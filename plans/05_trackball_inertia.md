# トラックボール慣性スクロール

## 目的

ゆっくりしたスクロールの精密さは保ちつつ、速いフリックが終わった後だけ縦・横スクロールを自然に減速継続させる。

通常カーソル、Slow Mouse、オートマウス、トラックボールジェスチャー、クリックには慣性を適用しない。

現在は `scroll_win`（Layer 7、Windowsモード）にのみ適用する。`scroll_mac`（Layer 6）は慣性導入前の設定（`zip_xy_transform (INPUT_TRANSFORM_X_INVERT)` のみ、`zip_scroll_inertia` 参照なし）に戻してあり、慣性は適用しない。Mac側のスクロール慣性は Mac Mouse Fix（macOS用のサードパーティ製ユーティリティ）に委ねる方針とし、ファームウェア側では扱わない。

## 入力経路

スクロール時の処理順は次のとおり。

1. `zip_xy_transform`
2. `zip_xy_scaler`
3. `zip_xy_to_scroll_mapper`
4. `zip_scroll_snap`
5. `zip_scroll_inertia`
6. ZMK HIDスクロールレポート

`zip_scroll_inertia` は、倍率変換と方向固定を終えた `REL_HWHEEL` / `REL_WHEEL` だけを観測する。慣性によるスクロールはプロセッサ自身を仮想入力デバイスとして `input_report_rel()` で発行し、専用の `inertia_listener` からHIDへ送る。

この専用経路により、慣性イベントがtransform、scaler、mapper、scroll-snapを再度通ることを防ぐ。

## macOSの制約とレートフロア設計（背景・Windows経路にも適用される設計）

macOSはサードパーティ製HIDマウスのResolution Multiplier（HIDの高解像度ホイール値ネゴシエーション機構、Linuxの `REL_WHEEL_HI_RES` に相当）を実質サポートしない。そのため `CONFIG_ZMK_POINTING_SMOOTH_SCROLLING` を有効にしていても、Mac上では慣性が発行するスクロール値は常に**整数クリック単位**（1クリック＝macOS側の「スクロール速度」設定で決まる行数）としてしか解釈されない。Windowsは同機構をサポートするため、1カウントがさらに細かい単位（既定16分の1ノッチ）に解釈され、この制約を受けない。

この制約はMac経路固有の問題であり、レートフロア方式（下記）で緩和を試みたが、実機調整の難しさ（macOS側のスクロール速度設定を約1/3に下げる必要があるなど）から、最終的に**Mac経路では慣性自体を使わない**判断とした。ただしレートフロア方式のプロセッサ実装自体は、離散クリック単位に限らず一般的な「疎になる前に打ち切る」設計として妥当なため、**Windows経路（`scroll_win`）ではそのまま採用している**。

離散クリックしか出せない環境では、「動きが連続に見える」条件は**クリック間隔が人の知覚限界（目安100ms）を下回り続けること**に尽きる。減速を穏やかにするほど速度が下がる過程が長引き、間隔が開いた単発クリックが連なる区間が伸びて、そこがカクつきとして知覚される。したがって取るべき対策は「滑らかに減速させる」ことではなく、**密なクリック間隔を保てる速度域だけを使い、それを割り込む前に慣性を打ち切る**ことである。これを実現するのが `max-click-interval-ms`（レートフロア）。

- 判定は**非正規化（実際にホストへ送られるpost-scalerドメイン）の速度**で行う。知覚限界は実クリック間隔そのもので決まり、経路ごとにscalerが異なると同じ正規化速度でも実クリックレートが変わるため、正規化ドメインでは判定できない。
- tick処理内で、摩擦適用後の実速度が「これ以上遅いとクリック間隔が `max-click-interval-ms` を超える」しきい値（`min-velocity-fp`、コンパイル時に `1,000,000 / max-click-interval-ms` として算出）を下回ったら即座に停止する（`stop-speed` 判定と並列のOR条件）。
- 発火（arm）判定にも同じフロアを適用する。適用しないと、フロア未満の弱いフリックで「armされるが1tick目で即停止する」無駄なスケジューリングが常態化するため。
- 旧実装にあった「ソフト終端フェーズ」（`tail-speed`/`tail-friction-permille` によるtail区間の緩やかな減衰）は撤廃した。離散クリック出力の経路ではtail域が丸ごとレートフロアの下に沈むため意味をなさず、そもそも「滑らかに減速する」という発想自体が離散クリック出力とは相性が悪かった。tailが対処しようとしていた「疎なクリックのブツ切れ感」は、レートフロアが「まだ密な間隔で鳴っている最中に打ち切る」ことで症状ごと解消する。

## 速度正規化とスケーリング

`zip_scroll_inertia` はプロセッサの2セル引数で、直前の `zip_xy_scaler` が生成する処理済み速度を共通の基準へ正規化してから発火判定する。scalerを変更する場合は、慣性プロセッサの第2引数も同じ値へ揃える。

| 経路 | scaler | 慣性プロセッサ | 用途 |
|---|---:|---:|---|
| Windows | `40 / 100` | `<&zip_scroll_inertia 100 40>` | 通常（`scroll_win`, Layer 7） |
| Windows split | `2 / 100` | `<&zip_scroll_inertia 100 2>` | 既存倍率を維持 |

Mac経路（`scroll_mac`, Layer 6）は慣性を使わないため `zip_scroll_inertia` を参照しない。scalerは慣性導入前の `2/100` のまま、方向も `INPUT_TRANSFORM_X_INVERT` のみ（慣性導入時に一時的に加えていた `INPUT_TRANSFORM_Y_INVERT` も併せて取り消した）。split側の方向設定は変更しない。

## 動作

- `start-speed` 未満のゆっくりした操作では慣性を開始しない。
- 実際の post-scaler 速度が `max-click-interval-ms` から導かれるレートフロアを下回る場合も、慣性を開始・維持しない（前節参照）。
- 実スクロールが来るたびに保留中の慣性tickを無効化し、最新サンプルから速度を再評価する。
- scalerが生成する値0のサンプルは、発動待ちの慣性や速度計算を変更しない。
- 逆方向へ `reverse-cancel-threshold` 以上動かすと以前の慣性を即座に破棄する。
- `start-delay-ms` の間、実入力が来なければ慣性を開始する。
- 一度 `start-speed` を超えた後は、`stop-speed` 未満になるか、レートフロアを割り込むまで発動状態を維持する。
- `tick-ms` ごとに `friction-permille / 1000` を速度へ掛け、`stop-speed` 未満、またはレートフロア未満で終了する。
- 停止時は減衰済みの端数を送信せず破棄する。停止直前の単発スクロールを防ぐためである。
- セミコロンキーを離しても、慣性は自然停止するまで継続する。
- 速度と移動端数は固定小数点で保持し、低速なスクロールでも整数丸めによる早期停止を防ぐ。

## 初期設定

設定場所は `boards/shields/torabo_tsuki_lp/torabo_tsuki_lp.dtsi` の `zip_scroll_inertia`。

| プロパティ | 初期値 | 意味 |
|---|---:|---|
| `start-speed` | 10 counts/s | 慣性を開始する正規化後の最低速度。ごく低速のスクロールから慣性を開始する |
| `stop-speed` | 1 count/s | 慣性を終了する正規化後の速度 |
| `max-speed` | 1000 counts/s | 取り込む正規化後速度の上限。慣性開始直後の速さをさらに抑える |
| `friction-permille` | 980 | 1 tick後に残す速度（1000分率）。レートフロアが下限を管理するため、従来より緩やかな減衰にして密なクリック域での滑走を延ばす |
| `tick-ms` | 8 ms | 慣性レポートの周期 |
| `start-delay-ms` | 12 ms | 実入力停止から慣性開始までの待ち時間 |
| `min-dt-ms` | 4 ms | 同一ミリ秒の入力で速度が無効になることを防ぐ最低計算間隔 |
| `sample-timeout-ms` | 50 ms | 速度計算をリセットする入力間隔 |
| `reverse-cancel-threshold` | 1 count | 逆入力キャンセルの最低変位 |
| `max-click-interval-ms` | 100 ms | 実クリック間隔がこれを超える速度域に落ちたら慣性を打ち切る（レートフロア。前節参照） |

検算（Windows、scaler 40/100、cap一杯のフリック）: 実クリックレート上限400クリック/s（正規化1000×0.4）から、レートフロア（実10クリック/s = 間隔100ms）に達するまで滑走し、全区間でクリック間隔が2.5msから100msへ単調に増えるのみで、それを超える疎な区間は発生しない。

## 実機調整

1. 細かい操作でも発動する場合は `start-speed` を上げ、フリックしても発動しない場合は下げる。
2. 滑りすぎる場合は `friction-permille` を下げ、短すぎる場合は上げる。
3. フリック直後の距離が大きすぎる場合は `max-speed` を下げる。
4. 止まり際がまだ硬い/唐突に感じる場合は `max-click-interval-ms` を100→120程度へ上げる（フロアを緩め、より疎な間隔まで許容する）。

一度に複数の値を変えず、この順番で一項目ずつ実機確認する。
